Деревья поведения

Behaviour Trees (BT) — модульный формализм из робототехники и геймдева

Зачем ещё один формализм

FSM для сложного поведения (робот, NPC в игре) быстро превращается в «спагетти из стрелок»: каждое новое поведение требует переходов из каждого состояния. Добавил «подзарядку батареи»? Рисуй переход из всех 15 состояний.

Деревья поведения решают эту проблему через модульность: поведения — независимые поддеревья, которые можно добавлять, убирать и комбинировать без переписывания остального.

Как работает BT

BT — это дерево, которое обходится (tick) с заданной частотой (например, 10 Гц). Каждый узел при tick возвращает один из трёх статусов:

SUCCESS

Задача выполнена

FAILURE

Задача не удалась

RUNNING

Задача в процессе

Типы узлов

Составные (composite) — управляют потоком

→ Sequence (последовательность)

Выполняет детей слева направо. Если ребёнок вернул SUCCESS — переходит к следующему. Если FAILURE — сразу возвращает FAILURE. Если RUNNING — возвращает RUNNING.

Аналогия: логическое AND. «Подойти к двери И открыть И пройти» — если что-то не вышло, вся последовательность провалена.

? Fallback / Selector (выбор)

Выполняет детей слева направо. Если ребёнок вернул FAILURE — переходит к следующему. Если SUCCESS — сразу возвращает SUCCESS. Если RUNNING — возвращает RUNNING.

Аналогия: логическое OR. «Открыть ключом ИЛИ взломать ИЛИ найти другой вход» — пробуем альтернативы.

⇉ Parallel (параллельный)

Выполняет всех детей одновременно. Политика успеха: все SUCCESS (AND), хотя бы один SUCCESS (OR), или M из N.

Пример: «двигаться к цели» параллельно с «следить за препятствиями».

Листовые — выполняют работу

Action (действие)

Выполняет конкретное действие: вращать мотор, отправить сообщение, обновить переменную.

Возвращает SUCCESS по завершении, RUNNING пока работает, FAILURE при ошибке.

Condition (проверка)

Проверяет условие: батарея > 20%? Цель видна? Путь свободен?

Мгновенная: SUCCESS если истина, FAILURE если ложь. Никогда не RUNNING.

Декораторы — модифицируют поведение ребёнка

ДекораторЧто делает
InverterИнвертирует результат: SUCCESS ↔ FAILURE
Repeat(N)Повторяет ребёнка N раз
RetryUntilSucceedПовторяет, пока не SUCCESS
Timeout(T)FAILURE если ребёнок не завершился за T секунд
ForceSuccessВсегда возвращает SUCCESS (игнорирует FAILURE)

Пример: робот-уборщик

? Root (Fallback)
  ├─ → Убрать комнату (Sequence)
  │  ├─ 🔍 Комната грязная?
  │  ├─ ? Найти мусор (Fallback)
  │  │  ├─ 🔍 Мусор в поле зрения?
  │  │  └─ 🎬 Исследовать комнату
  │  ├─ 🎬 Подъехать к мусору
  │  └─ 🎬 Подобрать мусор
  └─ → Зарядиться (Sequence)
     ├─ 🔍 Батарея < 20%?
     ├─ 🎬 Ехать к зарядке
     └─ 🎬 Заряжаться

Как это работает на каждом tick:

  1. Root (Fallback) пробует первого ребёнка — «Убрать комнату»
  2. Sequence проверяет: комната грязная? Если да — ищет мусор, едет, подбирает
  3. Если «Убрать комнату» вернул FAILURE (комната чистая) — Fallback пробует «Зарядиться»
  4. Sequence проверяет: батарея < 20%? Если да — едет заряжаться
  5. Если оба ребёнка FAILURE — робот просто ждёт (следующий tick попробует снова)

Модульность: чтобы добавить «протирать пыль», достаточно добавить ещё один ребёнок в Root Fallback. Никакие существующие переходы не меняются — в отличие от FSM.

BT vs FSM

FSMBehaviour Tree
СтруктураГраф (состояния + стрелки)Дерево (узлы + ветвление)
Добавление поведенияДобавить переходы из каждого состоянияДобавить поддерево
РеактивностьРеагирует на событияПерепроверяет условия каждый tick
МодульностьНизкая (состояния связаны)Высокая (поддеревья независимы)
Формальный анализХорошо развитРазвивается (safety, liveness через FTS)
Сложность восприятияЛегко при ≤10 состоянияхЛегко при любом размере (иерархия)

Формальная семантика

BT можно формализовать как Functional Transition System (FTS):

$$BT = (\mathcal{T}, r, f_\Delta)$$

  • $\mathcal{T}$ — множество поддеревьев
  • $r$ — корневой узел
  • $f_\Delta: \mathcal{T} \times \mathbb{R} \rightarrow \{S, F, R\}$ — функция обхода, возвращающая статус для каждого поддерева в зависимости от состояния мира

Это позволяет доказывать свойства безопасности и живости для BT, хотя формальная верификация BT — активная область исследований.

Инструменты

BehaviorTree.CPP

C++ библиотека, стандарт де-факто в ROS2. Используется в Nav2. XML-описание деревьев.

Groot

Визуальный редактор для BehaviorTree.CPP. Drag-and-drop, мониторинг в реальном времени.

py_trees

Python-библиотека. Используется с ROS1/ROS2 через py_trees_ros.

Unreal Engine BT

Встроенный BT-редактор в UE. Стандарт для AI персонажей в AAA-играх.

Дальше

BT — мощный инструмент для робототехники и игр, но в промышленной автоматике используют другой формализм. Переходим к GRAFCET / SFC — стандарту программирования ПЛК.
