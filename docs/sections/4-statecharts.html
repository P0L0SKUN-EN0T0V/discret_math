Statecharts (Харел)

Иерархические автоматы — FSM на стероидах, стандарт UML State Machine

Проблема, которую решают

FSM для реальной системы (например, бортовой компьютер автомобиля) потребует тысячи состояний. Граф станет нечитаемым. Причины:

  1. Нет иерархии — «двигатель.запущен.прогрев» и «двигатель.запущен.работа» — это разные состояния верхнего уровня
  2. Нет параллелизма — «двигатель × коробка × климат» = декартово произведение
  3. Нет истории — при возврате из подсостояния забывается, где были

В 1987 году Давид Харел (David Harel) предложил Statecharts — расширение FSM с тремя ключевыми механизмами: иерархия, ортогональные регионы и история.

Три ключевых механизма

1. Иерархия (вложенные состояния)

Состояние может содержать вложенные подсостояния. Переход во внешнее состояние автоматически переводит в его начальное подсостояние. Переход из внешнего состояния действует из любого подсостояния (не нужно рисовать переход из каждого).

Без иерархии нужно было бы рисовать переход fault из каждого подсостояния отдельно. При 20 подсостояниях — это 20 стрелок вместо одной.

2. Ортогональные регионы (параллелизм)

Состояние может содержать несколько параллельных регионов, разделённых пунктирной линией. Система находится в одном подсостоянии каждого региона одновременно.

3. Псевдосостояния «История»

При повторном входе в иерархическое состояние можно выбрать:

  • H (shallow history) — войти в то подсостояние верхнего уровня, в котором были при последнем выходе
  • H* (deep history) — войти точно в то подсостояние любого уровня вложенности, где были
  • Без истории — войти в начальное подсостояние (по умолчанию)

Пример: робот уходит в АВАРИЮ из РАБОТАЕТ.ПРОГРЕВ. После сброса аварии с H* — возвращается в ПРОГРЕВ, а не начинает заново.

Формальное определение

Statechart — это расширенный автомат:

$$SC = (S, \rho, \psi, T, s_0)$$

$S$Множество состояний (включая вложенные)
$\rho: S \rightarrow S$Функция родителя — кто в кого вложен (иерархия)
$\psi: S \rightarrow \{OR, AND, BASIC\}$Тип состояния: OR = обычная вложенность (одно активно), AND = параллельные регионы, BASIC = лист
$T$Множество переходов с событиями, условиями и действиями
$s_0$Начальная конфигурация

Каждый переход: $t = (source, target, event[guard]/action)$

Пример: ПРОГРЕВ → НОРМА : temp_ok[temp > 60°C] / log("прогрев завершён")

Действия: entry / exit / do

Каждое состояние может иметь три типа связанных действий:

ТипКогда выполняетсяПример
entry/При входе в состояниеentry/ включить_мотор()
exit/При выходе из состоянияexit/ выключить_мотор()
do/Пока система в состоянии (непрерывная активность)do/ контролировать_температуру()

Порядок при переходе A → B: exit(A) → action(перехода) → entry(B)

При иерархии: exit вызывается от глубокого к поверхностному, entry — от поверхностного к глубокому.

Преимущества над FSM

Проблема FSMРешение в StatechartsЭффект
Взрыв состояний при параллелизме Ортогональные регионы $k_1 + k_2$ вместо $k_1 \times k_2$
Дублирование переходов Переход из родительского состояния 1 стрелка вместо $n$

Инструменты

Stateflow (MATLAB)

Промышленный стандарт. Полная поддержка Statecharts, генерация C-кода, верификация.

YAKINDU / itemis CREATE

Бесплатный для некоммерческого использования. Графический редактор, генерация C/C++/Java.

XState (JavaScript)

Популярная библиотека для веба. Statecharts с визуализатором. Генерация из JSON.

UML State Machine (любой UML-редактор)

Enterprise Architect, StarUML, PlantUML. Statecharts стали частью стандарта UML 2.x.

Дальше

Statecharts решают проблемы иерархии и параллелизма для логики, но не моделируют время. Переходим к временным сетям Петри — расширению, которое добавляет задержки и позволяет анализировать производительность.
