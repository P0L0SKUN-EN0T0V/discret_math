Сети Петри — когда процессов больше одного

Параллелизм, ресурсы, синхронизация — всё то, что FSM не может

В чём проблема

В прошлой главе мы видели: FSM ломается, когда в системе несколько процессов работают одновременно. Количество состояний растёт как произведение, и при 10 компонентах мы уже имеем миллион состояний.

Но в реальности параллелизм — это норма, а не исключение. Два робота на конвейере. Три потока в программе. Десять станков на заводе. Нам нужен инструмент, который умеет описывать параллельные процессы естественно, без перечисления всех комбинаций.

Этот инструмент придумал немецкий математик Карл Адам Петри в 1962 году, в своей докторской диссертации. Ему было 23 года.

Аналогия: водопроводная сеть

Прежде чем лезть в формальности, представь водопроводную систему:

  • Есть резервуары — баки с водой. В каждом баке может быть 0, 1, 2... литров воды.
  • Есть клапаны — они соединяют резервуары.
  • Клапан открывается (срабатывает) только тогда, когда во всех подводящих резервуарах достаточно воды.
  • Когда клапан открывается, он забирает воду из подводящих резервуаров и наливает в отводящие.

Вот и всё. Сеть Петри — это именно это, только вместо «резервуаров» говорят места, вместо «клапанов» — переходы, а вместо «литров воды» — токены (фишки).

Токены — это ключевая штука. В FSM у тебя одна «фишка» — текущее состояние. В сети Петри фишек может быть много, и они разложены по разным местам. Каждая фишка — это свой процесс, свой ресурс, своя задача в очереди.

Три элемента сети Петри

Места (кружки)

Место — это «контейнер» для токенов. Оно описывает условие, состояние ресурса или этап процесса.

Примеры: «робот свободен», «деталь в буфере», «станок работает», «3 задачи в очереди».

Место с 2 токенами

Переходы (прямоугольники)

Переход — это «событие» или «действие». Он забирает токены из входных мест и кладёт в выходные.

Примеры: «взять деталь», «начать обработку», «отправить сообщение».

Дуги (стрелки)

Стрелки соединяют места с переходами и переходы с местами. Важное правило: стрелки идут только между разнотипными элементами. Место → переход или переход → место. Никогда место → место или переход → переход.

Стрелка «место → переход» означает: «это место — вход для этого перехода, переход забирает отсюда токены». Стрелка «переход → место» означает: «это место — выход, переход кладёт сюда токены».

Как это работает: правило срабатывания

Весь механизм сети Петри сводится к одному простому правилу:

Переход может сработать (он «разрешён»), если во всех его входных местах есть хотя бы по одному токену.

Когда переход срабатывает:

  1. Из каждого входного места забирается один токен
  2. В каждое выходное место кладётся один токен

И всё. Больше никаких правил. Вся мощь сетей Петри — из этого одного правила.

Давай проследим на простейшем примере:

Базовые паттерны: что умеют сети Петри

Из простого правила «забрал — положил» вырастают сложные поведения. Вот основные паттерны — запомни их, они будут встречаться повсюду.

Последовательность

Самое простое: цепочка мест и переходов. Токен идёт по цепочке слева направо, как в FSM.

p₁ → t₁ → p₂ → t₂ → p₃

Пока токен в p₁ — можно только t₁. После t₁ — токен в p₂, можно только t₂. Полный аналог конечного автомата.

Ветвление (выбор)

Из одного места выходят стрелки к нескольким переходам. Токен один — сработает только один из переходов. Какой именно — не определено. Это недетерминированный выбор.

t₁ ← p₁ → t₂
(токен в p₁: сработает либо t₁, либо t₂ — но не оба)

Пример из жизни: деталь на конвейере. Впереди развилка — налево на фрезу, направо на сверло. Деталь поедет только в одну сторону.

Параллелизм (разветвление / fork)

Один переход имеет несколько выходных мест. При срабатывании токены появляются во всех выходных местах одновременно. Так один процесс порождает несколько параллельных.

p₁ → t₁ → p₂
             ↘ p₃
(t₁ сработал: токен из p₁ исчез, появились токены в p₂ И в p₃)

Вот он, параллелизм! Теперь в системе два токена — два независимых процесса. FSM так не умеет.

Пример: заказ в ресторане отправлен. Одновременно: кухня начинает готовить (токен в «кухня работает») и бар начинает делать напиток (токен в «бар работает»).

Синхронизация (слияние / join)

Переход с несколькими входными местами. Он сработает только когда во ВСЕХ входных местах есть токены. Это «ждём самого медленного».

p₂ → t₂ ← p₃
(t₂ сработает только когда есть токены И в p₂, И в p₃)

Пример: блюдо и напиток готовы — только тогда официант несёт заказ. Если кухня закончила, а бар ещё нет — ждём.

В программировании: Promise.all(), pthread_join(), barrier. Всё это — паттерн «join» из сетей Петри.

Взаимное исключение (мьютекс)

Одно место с одним токеном, к которому подключены два перехода. Токен один — значит, сработать может только один из переходов. Второй будет ждать, пока токен вернётся.

Это и есть мьютекс. Место с одним токеном — это «ключ» или «пропуск». Кто взял — тот работает. Остальные ждут.

Пример: принтер в офисе. Один принтер (один токен в месте «принтер свободен»). Два сотрудника хотят печатать. Первый «забирает» токен (начинает печать), второй ждёт. Когда первый закончил — «возвращает» токен, второй может начать.

Живой пример: два робота и одна дрель

Теперь соберём всё вместе на конкретной задаче. Два робота работают на производственной линии. Каждый делает своё: берёт деталь, обрабатывает дрелью, кладёт обратно. Но дрель одна на двоих — одновременно её использовать нельзя.

Шаг 1: определяем места

  • p₁ — Робот A свободен (ждёт работы)
  • p₂ — Робот A работает с дрелью
  • p₃ — Робот B свободен
  • p₄ — Робот B работает с дрелью
  • p₅ — Дрель свободна

Шаг 2: определяем переходы

  • t₁ — Робот A берёт дрель. Входы: p₁ (робот свободен) + p₅ (дрель свободна). Выход: p₂ (робот работает)
  • t₂ — Робот A возвращает дрель. Вход: p₂. Выходы: p₁ + p₅
  • t₃ — Робот B берёт дрель. Входы: p₃ + p₅. Выход: p₄
  • t₄ — Робот B возвращает дрель. Вход: p₄. Выходы: p₃ + p₅

Шаг 3: начальная маркировка

«Маркировка» — это просто модное слово для «сколько токенов в каждом месте прямо сейчас». Начальная маркировка — это состояние при включении:

$M_0 = (p_1{:}1,\; p_2{:}0,\; p_3{:}1,\; p_4{:}0,\; p_5{:}1)$

Оба робота свободны (по токену в p₁ и p₃), дрель свободна (токен в p₅). Никто не работает (p₂ и p₄ пусты).

Шаг 4: прокручиваем в голове

Давай посмотрим, что произойдёт. В начальном состоянии какие переходы разрешены?

t₁ — нужны токены в p₁ и p₅. Есть? Да, оба на месте. Разрешён.

t₂ — нужен токен в p₂. Есть? Нет, p₂ пуст. Не разрешён.

t₃ — нужны токены в p₃ и p₅. Есть? Да. Разрешён.

t₄ — нужен токен в p₄. Нет. Не разрешён.

Итого: разрешены t₁ и t₃. Оба хотят токен из p₅ (дрель). Но токен в p₅ один! Значит сработает только один из них — кто первый, тот и забрал дрель.

Допустим, сработал t₁ (робот A взял дрель):

Новая маркировка: $M_1 = (p_1{:}0,\; p_2{:}1,\; p_3{:}1,\; p_4{:}0,\; p_5{:}0)$

Робот A работает (p₂=1), робот B свободен (p₃=1), но дрель занята (p₅=0).

Теперь t₃ не разрешён — в p₅ нет токена. Робот B вынужден ждать. Вот оно — взаимное исключение в действии!

Когда робот A закончит (t₂ сработает), токен вернётся в p₁ и p₅. Теперь t₃ разрешён — робот B может взять дрель.

Главный вывод: мы нигде не писали «если робот A работает, то робот B ждёт». Это поведение само возникает из структуры сети — из того, что p₅ содержит ровно один токен. Сеть Петри описывает не алгоритм, а правила игры, а поведение получается автоматически.

Сравнение: то же самое на FSM

Для контраста — как выглядит та же задача на конечном автомате?

Нужно перечислить все комбинации состояний двух роботов и дрели:

  1. (A свободен, B свободен, дрель свободна)
  2. (A работает, B свободен, дрель у A)
  3. (A свободен, B работает, дрель у B)

Тут всего 3 состояния — терпимо. Но добавь третьего робота — уже 7. Четвёртого — 15. Десятого... не хочется считать.

В сети Петри — просто добавляешь ещё два места (p₆, p₇) и два перехода (t₅, t₆) для нового робота. Размер сети растёт линейно, а не экспоненциально.

Формальная запись

Как и с FSM — сначала поняли идею, теперь запишем строго.

Сеть Петри — это четвёрка:

$$N = (P, T, F, W)$$

  • $P$ — конечное множество мест. Для нашего примера: $P = \{p_1, p_2, p_3, p_4, p_5\}$
  • $T$ — конечное множество переходов. $T = \{t_1, t_2, t_3, t_4\}$
  • $F$ — множество дуг (стрелок). Каждая дуга идёт либо от места к переходу, либо от перехода к месту: $F \subseteq (P \times T) \cup (T \times P)$
  • $W$ — вес дуги: сколько токенов она передаёт. Обычно $W = 1$ (по одному токену), но можно и больше

Плюс начальная маркировка $M_0: P \rightarrow \mathbb{N}$ — сколько токенов в каждом месте при старте.

Правило срабатывания записывается так:

Переход $t$ разрешён при маркировке $M$, если:

$$\forall p \in {}^\bullet t: \quad M(p) \geq W(p, t)$$

Здесь ${}^\bullet t$ — это «входные места» перехода $t$ (все места, из которых в $t$ идут стрелки). Запись $M(p) \geq W(p, t)$ означает: «токенов в месте $p$ не меньше, чем вес дуги из $p$ в $t$».

После срабатывания:

$$M'(p) = M(p) - W(p, t) + W(t, p)$$

Из входных мест убавилось, в выходные прибавилось. Если место не связано с переходом — ничего не меняется.

Что можно проверить

Сети Петри, как и FSM, позволяют задавать формальные вопросы. Но спектр вопросов шире — появляются новые, специфичные для параллельных систем:

Взаимное исключение

«Могут ли два робота одновременно держать дрель?» Формально: достижима ли маркировка, в которой $M(p_2) \geq 1$ и $M(p_4) \geq 1$? Если нет — взаимное исключение гарантировано.

В нашем примере ответ — нет, не достижима. Потому что p₂=1 и p₄=1 суммарно требуют, чтобы из p₅ забрали 2 токена, а у нас их всего 1. Это можно доказать формально через инварианты (об этом ниже).

Отсутствие тупиков (deadlock-freedom)

«Может ли система полностью застрять — так, что ни один переход не разрешён?» Если да — это тупик. Два процесса ждут друг друга, и оба стоят вечно.

Классический пример тупика: два робота, два инструмента. Робот A взял дрель и ждёт отвёртку. Робот B взял отвёртку и ждёт дрель. Оба стоят. Сеть Петри позволяет обнаружить это до запуска системы.

Ограниченность

«Не будут ли токены бесконечно накапливаться?» Если в каком-то месте число токенов может расти без предела — это аналог переполнения буфера или утечки памяти.

Сеть, в которой число токенов в каждом месте всегда ≤ $k$, называется $k$-ограниченной. Если $k = 1$ — она «безопасная» (safe). Наш пример с роботами — безопасная сеть.

Живость

«Гарантировано ли, что каждый переход сможет когда-нибудь сработать снова?» Если переход навсегда заблокирован — часть системы «мертва». Живая сеть — та, в которой каждый переход рано или поздно снова станет разрешённым.

Инварианты: доказательства «на салфетке»

Один из самых мощных приёмов анализа сетей Петри — инварианты. Это утверждения, которые верны при любом выполнении сети, вне зависимости от порядка срабатывания переходов.

Посмотри на нашу сеть с двумя роботами. Робот A либо свободен (токен в p₁), либо работает (токен в p₂). Третьего не дано. Значит:

$$M(p_1) + M(p_2) = 1 \quad \text{(всегда)}$$

Это инвариант. Его можно доказать, посмотрев на каждый переход: t₁ забирает из p₁ и кладёт в p₂ (сумма не меняется). t₂ забирает из p₂ и кладёт в p₁ (сумма не меняется). Все остальные переходы вообще не трогают p₁ и p₂.

Аналогично:

  • $M(p_3) + M(p_4) = 1$ — робот B либо свободен, либо работает
  • $M(p_2) + M(p_4) + M(p_5) = 1$ — дрель всегда «где-то»: либо у робота A, либо у робота B, либо свободна

Из последнего инварианта сразу следует: $M(p_2) + M(p_4) \leq 1$ — оба робота не могут одновременно работать с дрелью. Взаимное исключение доказано. Без перебора состояний, без моделирования — чистая математика.

Такие инварианты называются S-инварианты (place invariants). Они находятся как решения уравнения $\vec{y}^T \cdot C = 0$, где $C$ — матрица инцидентности сети. Но пугаться матриц не стоит — для небольших сетей инварианты видны невооружённым глазом.

Матричное представление (для любознательных)

Сеть Петри можно записать матрицей — это полезно для компьютерного анализа. Идея простая: для каждого перехода записываем, сколько токенов он забирает из каждого места и сколько кладёт.

Для нашего примера матрица инцидентности $C$ (строки — места, столбцы — переходы, значение = «сколько добавляется минус сколько забирается»):

$$C = \begin{pmatrix} -1 & 1 & 0 & 0 \\ 1 & -1 & 0 & 0 \\ 0 & 0 & -1 & 1 \\ 0 & 0 & 1 & -1 \\ -1 & 1 & -1 & 1 \end{pmatrix}$$

Например, столбец t₁: p₁ теряет 1 (−1), p₂ получает 1 (+1), p₅ теряет 1 (−1). Остальные не затронуты (0).

Чтобы найти маркировку после серии срабатываний:

$$M' = M_0 + C \cdot \vec{u}$$

где $\vec{u}$ — вектор «сколько раз сработал каждый переход». Если $\vec{u} = (1, 1, 0, 0)^T$ (робот A взял и вернул дрель), то $M' = M_0 + C \cdot \vec{u} = M_0$ — вернулись в начало.

Ограничения сетей Петри

Сети Петри решают проблему параллелизма и ресурсов. Но у них свои слабости:

  • Нет иерархии. Все места и переходы — на одном уровне. При 50+ элементах сеть становится нечитаемой. Для иерархии нужны Statecharts (глава 3).
  • Сложный анализ. Проверка достижимости в общем случае — задача огромной вычислительной сложности (EXPSPACE). Для больших сетей точный анализ может быть непрактичным.
  • Нет данных. Классические токены анонимны — они все одинаковые. Нельзя сказать «этот токен — деталь типа А, а тот — типа Б». Для этого есть расширение — цветные сети Петри.
  • Нет времени. Классическая сеть Петри не знает, сколько длится переход. Для анализа производительности нужны временные сети Петри (глава 4).

Что запомнить

  • Сеть Петри = места (кружки) + переходы (прямоугольники) + токены (фишки)
  • Переход срабатывает, когда во всех входных местах есть токены. Забирает из входов, кладёт в выходы
  • Базовые паттерны: последовательность, ветвление, fork (параллелизм), join (синхронизация), мьютекс
  • Параллелизм = несколько токенов в разных местах одновременно. Ресурс = место с ограниченным числом токенов
  • Инварианты позволяют доказывать свойства (взаимное исключение, сохранение ресурсов) математически
  • Проблемы: нет иерархии, нет данных, нет времени — для этого есть расширения

Дальше

Сети Петри справляются с параллелизмом, но не с иерархией. Когда состояний десятки — граф превращается в кашу. В главе 3: Statecharts разберёмся, как навести порядок с помощью вложенных состояний и параллельных регионов.
