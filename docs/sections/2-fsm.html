Конечные автоматы (FSM)

Finite State Machine — фундамент дискретного моделирования

Интуиция

Конечный автомат — это самая простая модель дискретной системы. Система находится в одном из конечного числа состояний. При получении входного сигнала она переходит в другое состояние (или остаётся в том же).

Аналогия: турникет в метро. Два состояния: ЗАБЛОКИРОВАН и РАЗБЛОКИРОВАН. Два входа: «жетон» и «проход». Поведение полностью определяется таблицей переходов.

ЗАБЛОК.
жетон →
← проход
РАЗБЛОК.

Жетон в заблокированном → разблокируется. Проход в разблокированном → блокируется обратно.

Формальное определение

Детерминированный конечный автомат (DFA) — это пятёрка:

$$\mathcal{A} = (Q, \Sigma, \delta, q_0, F)$$

$Q$Конечное множество состояний$\{q_0, q_1, q_2, \ldots\}$
$\Sigma$Конечный входной алфавит$\{a, b, c, \ldots\}$
$\delta$Функция переходов $Q \times \Sigma \rightarrow Q$$(q_0, a) \mapsto q_1$
$q_0$Начальное состояние $q_0 \in Q$
$F$Множество допускающих состояний $F \subseteq Q$

Детерминированный означает: для каждой пары (состояние, вход) существует ровно один переход. Никакой неоднозначности.

Виды конечных автоматов

По детерминизму

DFA — детерминированный

$\delta: Q \times \Sigma \rightarrow Q$

Из каждого состояния по каждому символу — ровно один переход. Однозначное поведение.

Используется в реальных системах управления, где поведение должно быть предсказуемым.

NFA — недетерминированный

$\delta: Q \times \Sigma \rightarrow 2^Q$

Из одного состояния по одному символу — несколько возможных переходов. Плюс $\varepsilon$-переходы (без входа).

Удобнее для моделирования, но всегда можно преобразовать в DFA (ценой роста числа состояний).

Теорема: для любого NFA существует эквивалентный DFA. Но число состояний DFA может быть до $2^{|Q_{\text{NFA}}|}$ — экспоненциальный взрыв. Пример: NFA с 5 состояниями может потребовать DFA с 32 состояниями.

По типу выхода

Автомат Мура

Выход зависит только от состояния:

$\lambda: Q \rightarrow \Gamma$

Каждому состоянию сопоставлен свой выходной сигнал. Выход меняется при смене состояния.

Пример: светофор. Состояние «КРАСНЫЙ» → выход «стоп». Состояние «ЗЕЛЁНЫЙ» → выход «можно ехать».

Автомат Мили

Выход зависит от состояния И входа:

$\lambda: Q \times \Sigma \rightarrow \Gamma$

Выходной сигнал привязан к переходу, а не к состоянию. Более компактные модели.

Пример: автомат оплаты — выход «звенит» только при переходе из «ожидание» в «оплачено» (т.е. в момент опускания монеты).

Мур vs Мили: эквивалентны по выразительности. Автомат Мура всегда можно превратить в автомат Мили и наоборот. Мили обычно компактнее (меньше состояний), Мура — проще для реализации (выход стабилен внутри состояния).

Реализация в коде

FSM — единственный формализм, который напрямую ложится в код. Три классических подхода:

1. Switch-case (самый простой)

enum State { IDLE, APPROACH, GRIP, LIFT, MOVE, PLACE, ERROR };
State state = IDLE;

void handle_event(Event e) {
    switch (state) {
        case IDLE:
            if (e == PART_DETECTED) {
                move_to_pickup();
                state = APPROACH;
            }
            break;
        case APPROACH:
            if (e == REACHED) {
                close_gripper();
                state = GRIP;
            }
            break;
        // ... аналогично для остальных
    }
    // Глобальный переход в ERROR
    if (e == FAULT) {
        emergency_stop();
        state = ERROR;
    }
}

Плюсы: понятно, быстро, нулевой оверхед. Минусы: при 20+ состояниях switch превращается в спагетти. Нет формальной верификации.

2. Таблица переходов (data-driven)

# Python: таблица переходов как словарь
transitions = {
    ('IDLE',     'part_detected'): ('APPROACH', move_to_pickup),
    ('APPROACH', 'reached'):       ('GRIP',     close_gripper),
    ('GRIP',     'gripped'):       ('LIFT',     lift_arm),
    ('LIFT',     'lifted'):        ('MOVE',     move_to_pallet),
    ('MOVE',     'arrived'):       ('PLACE',    open_gripper),
    ('PLACE',    'placed'):        ('IDLE',     return_home),
}

def handle(state, event):
    key = (state, event)
    if event == 'fault':
        return 'ERROR', emergency_stop
    if key in transitions:
        next_state, action = transitions[key]
        action()
        return next_state, action
    return state, None  # нет перехода — остаёмся

Плюсы: таблицу можно загрузить из файла, визуализировать, проверить на полноту. Минусы: чуть больше абстракции.

3. State Pattern (ООП)

class State:
    def handle(self, event, context): pass

class IdleState(State):
    def handle(self, event, context):
        if event == 'part_detected':
            context.move_to_pickup()
            return ApproachState()
        return self

class ApproachState(State):
    def handle(self, event, context):
        if event == 'reached':
            context.close_gripper()
            return GripState()
        return self
# ... каждое состояние — отдельный класс

Плюсы: каждое состояние инкапсулировано, легко добавлять новые. Минусы: много классов, переходы размазаны по файлам.

Анализ свойств FSM

Для конечного автомата можно формально проверить:

Полнота

Для каждой пары $(q, \sigma)$ определён переход? Если нет — система может «зависнуть», получив неожиданный вход.

Проверка: пройти по всем $|Q| \times |\Sigma|$ комбинациям. Сложность $O(|Q| \cdot |\Sigma|)$.

Достижимость

Все ли состояния достижимы из $q_0$? Недостижимое состояние — мёртвый код.

Проверка: BFS/DFS из $q_0$ по графу переходов. Сложность $O(|Q| + |\delta|)$.

Отсутствие тупиков

Есть ли состояния без исходящих переходов (кроме допускающих)? Тупик = система застряла.

В нашем примере: ERROR — не тупик, потому что из него есть переход по reset.

Минимизация

Можно ли уменьшить число состояний без потери поведения? Алгоритм Хопкрофта находит минимальный DFA за $O(n \log n)$.

Два состояния эквивалентны, если для любой входной цепочки они дают одинаковый выход.

Ограничения FSM

FSM — мощный инструмент, но у него есть фундаментальные ограничения:

  1. Взрыв состояний. Если система имеет $n$ независимых компонент с $k$ состояниями каждая, FSM потребует $k^n$ состояний (все комбинации). Два процесса по 10 состояний = 100 состояний в FSM.
  2. Нет параллелизма. FSM находится ровно в одном состоянии в каждый момент. Параллельные процессы нужно кодировать как декартово произведение — отсюда взрыв.
  3. Нет памяти. FSM не может считать (нет счётчиков, стеков). Он распознаёт ровно регулярные языки. Пример невозможного: «открыто ровно столько скобок, сколько закрыто» — для этого нужен как минимум автомат с магазинной памятью.
  4. Плоская структура. Нет иерархии, нет вложенности. При 50+ состояниях граф становится нечитаемым.

Эти ограничения мотивируют расширения: Statecharts решают проблемы 2 и 4, сети Петри — проблемы 1, 2 и 3.

Выразительность: иерархия Хомского

FSM распознают в точности регулярные языки (тип 3 по Хомскому). Это самый нижний уровень иерархии:

ТипГрамматикаАвтоматПример
3РегулярнаяFSM$a^*b$ — любое число $a$, затем $b$
2Контекстно-свободнаяАвтомат с магазинной памятью$a^n b^n$ — скобки
1Контекстно-зависимаяЛинейно ограниченный автомат$a^n b^n c^n$
0Рекурсивно перечислимаяМашина ТьюрингаВсё вычислимое

Сети Петри по выразительности находятся строго между типом 3 и типом 0 — они мощнее FSM, но слабее машины Тьюринга.

Дальше

FSM — фундамент, но для параллельных систем и ресурсов он слаб. Переходим к сетям Петри — формализму, который решает главные проблемы FSM: параллелизм, синхронизацию и ресурсы.
